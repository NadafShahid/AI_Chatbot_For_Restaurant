Restaurant API - Payment Management

This documentation covers the Payment Management APIs for the Restaurant Backend API. All endpoints are prefixed with /api/payments.

Base URL

http://localhost:3000/api/payments

Authentication

Currently, no authentication is required for these endpoints. In production, implement proper authentication middleware.

Payment Status Flow

Payments can have the following statuses:

  • pending → Payment initiated but not completed
  • completed → Payment successfully processed
  • failed → Payment failed or was declined
  • refunded → Payment was refunded

Order Status Integration

When payments are completed, the associated order status may be updated to reflect payment completion:

  • delivered → Order delivered and ready for payment
  • paid → Order payment completed successfully

Note: Order status updates are handled through the Order Management API endpoints.

Features

  • Record payments linked to orders
  • Validate payment amounts against order totals
  • Track payment status changes
  • Retrieve payments by order
  • Comprehensive payment history

API Endpoints

POST/api/payments

Record a payment for an order

Request Body

{ "order_id": 1, "amount": 28.97, "method": "online" }
FieldTypeRequiredDescription
order_idintegerYesOrder ID to link payment to
amountnumberYesPayment amount (must match order total)
methodstringYesPayment method (e.g., 'online', 'cash', 'card')

cURL Example

curl -X POST http://localhost:3000/api/payments \ -H "Content-Type: application/json" \ -d '{ "order_id": 1, "amount": 28.97, "method": "online" }'

Success Response (201)

{ "success": true, "message": "Payment recorded successfully", "data": { "id": 1, "order_id": 1, "amount": "28.97", "method": "online", "status": "pending", "created_at": "2024-01-15T10:35:00.000Z" } }

Error Responses

400 Bad Request: Missing required fields or amount doesn't match order total
404 Not Found: Order not found
500 Internal Server Error: Database error

GET/api/payments

Get all payments with order and user details

cURL Example

curl -X GET http://localhost:3000/api/payments

Success Response (200)

{ "success": true, "message": "Payments retrieved successfully", "data": [ { "id": 1, "order_id": 1, "amount": "28.97", "method": "online", "status": "completed", "created_at": "2024-01-15T10:35:00.000Z", "user_id": 1, "order_total": "28.97", "user_name": "John Doe" }, { "id": 2, "order_id": 2, "amount": "15.99", "method": "cash", "status": "pending", "created_at": "2024-01-15T11:00:00.000Z", "user_id": 2, "order_total": "15.99", "user_name": "Jane Smith" } ] }

GET/api/payments/order/:orderId

Get all payments for a specific order

URL Parameters

ParameterTypeRequiredDescription
orderIdintegerYesOrder ID

cURL Example

curl -X GET http://localhost:3000/api/payments/order/1

Success Response (200)

{ "success": true, "message": "Payments retrieved successfully", "data": [ { "id": 1, "order_id": 1, "amount": "28.97", "method": "online", "status": "completed", "created_at": "2024-01-15T10:35:00.000Z" } ] }

Error Responses

404 Not Found: Order not found
500 Internal Server Error: Database error

PUT/api/payments/:id/status

Update payment status

URL Parameters

ParameterTypeRequiredDescription
idintegerYesPayment ID

Request Body

{ "status": "completed" }
FieldTypeRequiredDescription
statusstringYesNew status (pending, completed, failed, refunded)

cURL Example

curl -X PUT http://localhost:3000/api/payments/1/status \ -H "Content-Type: application/json" \ -d '{ "status": "completed" }'

Success Response (200)

{ "success": true, "message": "Payment status updated successfully", "data": { "id": 1, "order_id": 1, "amount": "28.97", "method": "online", "status": "completed", "created_at": "2024-01-15T10:35:00.000Z", "user_id": 1, "order_total": "28.97", "user_name": "John Doe" } }

Error Responses

400 Bad Request: Invalid status
404 Not Found: Payment not found
500 Internal Server Error: Database error

  6. Verify payment-order linkage

Common Response Format

All API responses follow this structure:

{ "success": boolean, // true for success, false for error "message": string, // Response message "data": object/array // Response data (only on success) }

Payment Data Structure

The payment data includes:

  • id: Unique payment identifier
  • order_id: Linked order ID
  • amount: Payment amount (string format)
  • method: Payment method used
  • status: Current payment status
  • created_at: Payment creation timestamp
  • user_id: User who made the payment (in detailed responses)
  • order_total: Total amount of linked order
  • user_name: Name of user who made the payment

Error Handling

  • 400 Bad Request: Missing or invalid request data, amount mismatch
  • 404 Not Found: Order or payment not found
  • 500 Internal Server Error: Server/database errors

Notes

  • Payment amounts must exactly match the order total
  • Payments are linked to orders and cannot be created without a valid order
  • Payment status can be updated to track payment processing
  • All monetary values are returned as strings for precision
  • Multiple payments can be recorded for the same order if needed
  • Consider adding payment gateway integration in production
  • Implement proper user authentication and authorization
  • Add payment validation and fraud detection for production use
